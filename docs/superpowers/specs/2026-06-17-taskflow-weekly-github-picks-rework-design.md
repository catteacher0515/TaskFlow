# TaskFlow 每周 GitHub 精选模板改版规格

日期：2026-06-17

## 目标

把 `每周 GitHub 精选` 从“通用 Todo + 预置推荐位”的初版，收敛成更贴合真实周刊工作流的结构：

1. 候选仓库直接在 `亲测候选仓库` 中录入和处理。
2. 每个候选仓库支持 `入选 / 淘汰 / 暂缓` 三种结果，并允许改判。
3. `确定本周 5 个推荐` 不再手动维护，而是自动汇总所有被标记为 `入选` 的候选仓库。
4. `发布` 直接预置平台子任务，方便逐个平台打勾推进。

这次改版只处理周刊模板和周刊项目的详情交互，不改变通用任务模板和非周刊项目的行为。

## 范围

### 本次包含

1. 修改 `每周 GitHub 精选` 模板的默认任务骨架。
2. 修改周刊项目详情页里周刊特定区块的交互。
3. 为周刊候选仓库增加 `入选 / 淘汰 / 暂缓` 三种结果态。
4. 把 `确定本周 5 个推荐` 改为只读自动汇总区。
5. 为 `发布` 预置平台任务。
6. 为已有周刊项目提供兼容迁移，避免旧项目进入不可编辑状态。

### 本次不包含

1. 不为周刊重新引入独立的 `progressObject / slot / stage` 数据模型。
2. 不限制 `入选` 数量，允许超过 `5` 个。
3. 不做“本周用哪几个、剩下留到下周”的额外调度逻辑。
4. 不改动普通 Todo 项目的详情页表现。
5. 不新增专门的周刊发布统计、筛选或排序功能。

## 目标结构

周刊模板的一级区块固定为：

1. `亲测候选仓库`
2. `确定本周 5 个推荐`
3. `成稿`
4. `发布`

默认删除以下旧结构：

1. `收集候选仓库`
2. `写推荐理由`
3. `确定本周 5 个推荐` 下面的预置 `推荐 1` 到 `推荐 5`

`发布` 默认预置以下子任务：

1. `抖音`
2. `知乎`
3. `B站`
4. `小红书`
5. `编程导航`
6. `稀土掘金`

## 数据建模

继续复用现有 `taskTree`，不新增单独的候选仓库实体。

### 一级区块角色

1. `亲测候选仓库`
   - 是唯一的候选录入区。
   - 用户新增的候选仓库都作为它的直接子任务保存。

2. `确定本周 5 个推荐`
   - 是自动汇总区。
   - 不接受手动添加子任务。
   - 显示内容来自 `亲测候选仓库` 下所有结果为 `入选` 的候选项。

3. `成稿`
   - 保持普通 Todo 行为。

4. `发布`
   - 保持普通 Todo 行为。
   - 只是在默认骨架里自带 6 个平台子任务。

### 候选结果态映射

周刊候选仓库仍然使用任务节点保存，但在 `亲测候选仓库` 这个区块下使用专门的状态语义：

1. `completed` 表示 `入选`
2. `dropped` 表示 `淘汰`
3. `unhandled` 表示 `暂缓`
4. `not_started` 或 `active` 表示尚未定论

这样做的原因：

1. 可以复用已有 `taskTree`、任务状态接口和反馈逻辑。
2. 可以允许候选项后续改判，而不必新建或删除数据。
3. `确定本周 5 个推荐` 可以由任务状态派生，不需要额外存储一份推荐列表。

## 详情页交互

### 亲测候选仓库

`亲测候选仓库` 的交互规则：

1. 保留输入框，用于直接添加候选仓库。
2. 新增候选后，任务默认是未定状态。
3. 每个候选项提供三个明确动作：
   - `入选`
   - `淘汰`
   - `暂缓`
4. 三个结果态之间允许反复改判。

显示上不再把候选项仅仅当作普通待办勾选，而是明确显示它当前属于：

1. `未定`
2. `入选`
3. `淘汰`
4. `暂缓`

### 确定本周 5 个推荐

这个区块改成只读自动汇总：

1. 不显示“添加小任务”输入框。
2. 不允许手动新建、删除、编辑推荐项。
3. 只展示当前所有被标成 `入选` 的候选仓库。
4. 某个候选从 `入选` 改成 `淘汰` 或 `暂缓` 时，它会自动从这个区块消失。
5. 超过 `5` 个入选不报错、不阻止、不额外提示。

### 成稿与发布

1. `成稿` 保持普通 Todo。
2. `发布` 保持普通 Todo，但默认带平台子任务。
3. 这些区块继续使用现有的打勾、开始、添加子任务能力。

## 反馈与状态语义

本次不新增新的反馈种类，只复用现有任务反馈能力：

1. 候选仓库被标成 `入选` 时，仍按任务完成类行为处理。
2. 候选仓库被标成 `淘汰` 或 `暂缓` 时，不额外新增专属反馈类型。
3. 如果候选仓库后续改判，界面状态以最新结果为准。

这次改版优先保证流程顺畅，不扩展额外反馈语义。

## 迁移策略

### 已经是新周刊 Todo 骨架的项目

如果项目已经有 `taskTree`：

1. 删除 `收集候选仓库`
2. 删除 `写推荐理由`
3. 删除 `确定本周 5 个推荐` 下的预置 `推荐 1` 到 `推荐 5`
4. 给 `发布` 补齐 6 个平台子任务
5. 如果 `收集候选仓库` 下面已有你手工添加过的候选仓库，则移动到 `亲测候选仓库` 下

### 仍是旧阶段/槽位/推进对象结构的项目

如果项目是 `weekly-github-picks` 且 `taskTree` 缺失或为 `null`：

1. 自动生成新的周刊骨架
2. 读取旧的 `progressObjects`
3. 把每个旧候选仓库迁成 `亲测候选仓库` 下的一个子任务
4. 旧状态映射为：
   - `selected -> 入选`
   - `rejected -> 淘汰`
   - `maybe -> 暂缓`
   - 其他状态 -> 未定
5. 旧的 `stages / slots / progressObjects` 在归一化后不再参与详情渲染

这样旧项目即使最初是用旧进程创建的，也不会继续保持“不可编辑”状态。

## 验收标准

1. 新建 `每周 GitHub 精选` 项目后，默认只出现 4 个一级区块：
   - `亲测候选仓库`
   - `确定本周 5 个推荐`
   - `成稿`
   - `发布`

2. 页面中不再出现：
   - `收集候选仓库`
   - `写推荐理由`
   - `推荐 1-5`

3. `发布` 默认带：
   - `抖音`
   - `知乎`
   - `B站`
   - `小红书`
   - `编程导航`
   - `稀土掘金`

4. 在 `亲测候选仓库` 下新增一个候选仓库后，可以把它标成：
   - `入选`
   - `淘汰`
   - `暂缓`

5. 候选仓库结果态允许反复改判。

6. 被标成 `入选` 的候选仓库，会自动显示在 `确定本周 5 个推荐` 区块中。

7. 某候选从 `入选` 改成 `淘汰` 或 `暂缓` 后，会自动从 `确定本周 5 个推荐` 中移除。

8. `确定本周 5 个推荐` 不允许手动添加推荐项。

9. 对旧周刊项目重新读取后，不会再出现“项目能打开但没有任何可编辑内容”的状态。
