# TaskFlow 情绪页面规格

日期：2026-06-20

## 目标

为 TaskFlow 增加一个极轻量的 `情绪` 板块，用来承接“每天结束时，对这一天做一个简单情绪总结”这件事。

第一版要解决的问题：

1. 让用户每天结束时，能顺手为这一天落下一条情绪总结。
2. 让用户以后回看一个月时，能先通过一整个月的 emoji 变化，快速感知心路历程。
3. 保持低负担，不把它做成复杂日记系统。
4. 和 Pocket 吐槽形成配合：
   - Pocket 负责白天过程里的随手记录。
   - `情绪` 负责当天收口时的简短总结。

## 产品定位

`情绪` 是一个独立页面，和 `工作台 / 新建任务 / 习惯 / 模板 / 反馈` 同级。

它不是：

1. 项目反馈的一部分。
2. 习惯打卡的一部分。
3. 一个普通时间流日记工具。

它的核心语义是：

- 每天结束时，用一个必填 emoji 给这一天定一个情绪基调。
- 如有需要，再补一句短句或更详细的内容。

## 范围

### 本次包含

1. 顶部导航新增 `情绪` 页面入口。
2. 支持严格 `一天一条` 情绪记录。
3. 情绪记录包含三个可见字段：
   - `emoji`，必填
   - `shortNote`，选填
   - `detail`，选填
4. 情绪页面支持两种视图：
   - `月历`
   - `列表`
5. 默认优先进入当前月份的 `月历` 视图。
6. 支持查看和编辑任意日期的记录。
7. `detail` 默认折叠。

### 本次不包含

1. AI 总结。
2. 情绪统计图或趋势分析。
3. 标签系统。
4. 多条/天的情绪时间流。
5. 和 Pocket 的自动同步。
6. 删除记录或清空为未记录状态。

## 核心约束

### 一天一条

每个自然日只能有一条情绪记录。

如果这一天已经有记录，后续再次保存的行为是更新，而不是新增第二条。

### 不允许清空

一旦某天已保存，不允许把这一天恢复为“未记录”状态。

原因是：

- 用户认为“当天落下的状态就是当时最真实的状态”。
- 第一版不希望把情绪页面做成可反复撤销、反复美化历史的系统。

### emoji 是最小闭环

`emoji` 是每条记录必须完成的部分。

`shortNote` 和 `detail` 都是补充，而不是主骨架。

## 信息架构

顶部导航新增：

- `情绪`

情绪页面内部包含三块：

1. `月份切换区`
2. `视图切换区`
   - `月历`
   - `列表`
3. `日期编辑区`

第一版不拆成多个二级页面，全部放在同一个页面内完成。

## 记录模型

每条情绪记录至少包含以下字段：

- `date`
- `emoji`
- `shortNote?`
- `detail?`
- `createdAt`
- `updatedAt`

### date

表示这条记录归属的自然日，格式使用 `YYYY-MM-DD`。

### emoji

必填。

这是这一天的主情绪标签，也是月历视图里的核心展示信息。

### shortNote

选填。

用于记录一句非常短的总结，例如：

- `今天终于推进起来了`
- `卡得很厉害`
- `状态比预想稳`

它服务于列表视图，不承担长内容职责。

### detail

选填。

用于记录更详细的补充说明，例如对这一天的背景、原因、波动过程的描述。

第一版里它默认折叠，避免录入界面显得太重。

## emoji 选项

第一版采用固定少量选项，避免选择时过度犹豫。

固定选项如下：

1. `😭 很差`
2. `😞 低落`
3. `😐 平平`
4. `🙂 还行`
5. `😄 开心`
6. `😎 很顺`
7. `🤯 爆炸`

### 选择时的展示规则

在录入或编辑时，选项展示为：

- `emoji + 含义`

例如：

- `😄 开心`
- `🤯 爆炸`

这样可以帮助用户在选择时快速判断，不需要重新回忆每个 emoji 的含义。

### 展示时的规则

在月历和列表等最终展示层，不附带含义文字，只展示 emoji 本身。

原因是：

1. 月历视图要尽量干净。
2. 未来回看时，用户想先接收直观的情绪轨迹，而不是解释文本。

## 页面视图

### 月历视图

这是第一版默认视图，也是主视图。

目标是让用户一眼看到“这个月每天的情绪变化”。

#### 展示规则

1. 每天一个格子。
2. 格子主内容只展示 `emoji`。
3. 不在月历格子里展示 `shortNote` 或 `detail`。
4. 已选中的日期需要有明确高亮。
5. 未记录日期显示为空状态。

#### 交互规则

1. 点击某一天，同页的 `日期编辑区` 切换到这一天。
2. 支持切换当前查看月份。
3. 第一版以月为主要浏览单位，不做周视图。

### 列表视图

列表视图用于更线性地回看近期记录。

#### 展示规则

1. 只展示当前查看月份内的记录，并按日期倒序。
2. 如果该记录写了 `emoji + shortNote`，则展示：
   - `emoji + shortNote`
3. 如果没有填写 `shortNote`，则只展示：
   - `emoji`
4. `detail` 不在列表主展示位默认展开。

#### 交互规则

1. 点击某条记录，可以让 `日期编辑区` 切换到该日期。
2. 切换查看月份时，列表内容同步切换到该月份。

## 日期编辑区

`日期编辑区` 是这页唯一的录入入口。

它需要支持：

1. 为今天新增记录。
2. 为过去某一天补记记录。
3. 编辑某一天已有记录。

### 字段顺序

固定顺序：

1. 日期
2. emoji 选择
3. shortNote
4. detail（默认折叠）

### 编辑规则

1. 如果该日期还没有记录，则显示为新建态。
2. 如果该日期已有记录，则显示为编辑态。
3. 保存后直接覆盖该日期的记录。
4. 不提供删除按钮。
5. 不提供“清空为未记录”按钮。

## 日期查看能力

虽然产品主轴是“月历优先”，但仍需要支持自定义日期查看。

第一版至少应支持：

1. 在月历中切换到任意月份并点击具体日期。
2. 在编辑区直接选择一个日期。
3. 在列表中点击某条记录回到对应日期。

这样既能满足“按月回看”，也能满足“我想直接看某一天”的需求。

## 与现有系统的边界

### 与 Pocket 的关系

Pocket 负责白天随手吐槽和过程记录。

`情绪` 页面负责晚些时候对这一天做一个收口总结。

第一版只定义这种配合关系，不实现自动联动。

### 与反馈页的关系

反馈页记录的是推进结果，例如：

- 项目完成
- 任务完成
- 熵减
- 槽位填充

`情绪` 页面记录的是主观感受。

两者不应该混在同一个时间流里。

### 与习惯页的关系

习惯页解决的是“重复行为有没有做”。

`情绪` 页面解决的是“这一天整体是什么感觉”。

两者语义不同，不能合并。

## 成功标准

这版如果要算成功，至少要满足：

1. 用户能在 10 秒内完成当天的最小记录。
2. 用户回看一个月时，能一眼感知情绪波动轨迹。
3. 录入过程不会因为 emoji 太多或字段太重而产生明显犹豫。
4. 页面不会和 `反馈 / 习惯 / 项目` 的语义混淆。

## 后续观察点

第一版上线后，重点观察：

1. 7 个 emoji 是否足够表达真实波动。
2. 用户是否经常填写 `shortNote`，还是只写 emoji。
3. `detail` 是否真的需要保留，还是会长期闲置。
4. 是否有按周或按月聚合回顾的进一步需求。
