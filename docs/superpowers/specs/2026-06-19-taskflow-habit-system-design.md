# TaskFlow 习惯系统规格

日期：2026-06-19

## 目标

为 TaskFlow 增加一套独立于项目系统的轻量习惯系统，用来承接那些需要重复发生、但不适合放进项目待办清单的行为。

第一版习惯系统要解决的问题：

1. 把“交付物推进”和“重复行为维持”分开。
2. 让用户能单独管理每日或每周重复出现的小行为。
3. 让用户打开页面时，能先看到今天该做什么。
4. 支持晚一点补打卡，贴合真实使用方式。

## 产品定位

习惯系统是一个独立页面，和 `工作台 / 新建任务 / 模板 / 反馈` 同级。

它不属于任何项目，也不进入项目列表、项目详情或项目任务树。

边界定义如下：

- 项目：有明确交付物，需要推进到完成。
- 习惯：会反复出现，单次只关心“做了没”，重点是持续发生。

## 范围

### 本次包含

1. 顶部导航新增 `习惯` 页面入口。
2. 习惯页面提供两个主视角：
   - `今天`
   - `全部习惯`
3. 习惯支持按每周执行日配置。
4. 习惯支持两种周期：
   - `长期进行`
   - `有开始日期和结束日期`
5. 习惯记录只支持 `已完成 / 未完成`。
6. 到执行日未操作时，默认视为 `未完成`。
7. 支持后续回补为 `已完成`。
8. 支持归档习惯。

### 本次不包含

1. 标签系统。
2. 提醒系统。
3. 次数、分钟数、条数等数量型记录。
4. 连续打卡天数。
5. 热力图、统计图、评分。
6. 习惯自动进入项目待办清单。
7. 把习惯自动转换成项目或任务树节点。

## 信息架构

顶部导航新增：

- `习惯`

习惯页面内部提供两个视角切换：

1. `今天`
2. `全部习惯`

第一版不额外做独立的 `归档` 页面；归档内容先放在 `全部习惯` 视角内管理。

## 今天视角

`今天` 是习惯页面默认打开的第一屏。

页面分为两组：

1. `今天该做`
2. `历史漏项`

### 今天该做

- 展示今天按计划应执行的习惯。
- 每条习惯只提供最核心的动作：
  - `标记已完成`
- 第一版不在这一区展示复杂编辑入口。

### 历史漏项

- 展示过去应执行但目前仍为 `未完成` 的记录。
- 只展示仍处在该习惯有效周期内的漏项。
- 用户可以把这些历史漏项补改为 `已完成`。

`今天` 视角承担“执行”和“补打卡”职责，不承担复杂管理。

## 全部习惯视角

`全部习惯` 用于查看和管理习惯本身，而不是某一天的执行结果。

至少支持：

1. 查看当前所有未归档习惯。
2. 新建习惯。
3. 编辑习惯。
4. 归档习惯。
5. 查看已归档习惯。

第一版不做复杂筛选和多维管理。

## 习惯模型

每个习惯至少包含以下字段：

- `id`
- `title`
- `schedule`
- `period`
- `archivedAt?`
- `createdAt`
- `updatedAt`

### title

- 习惯名称，例如：
  - `爬坡`
  - `用 Pocket 3 记录当天`
  - `看 AI HOT 日报`

### schedule

用于描述每周哪些天执行。

第一版统一按“周几集合”建模，例如：

- 每天
- 周一到周五
- 周二、周四、周六

不额外拆成不同频率类型。

### period

支持两种形式：

1. `ongoing`
   - 长期进行
2. `bounded`
   - 有开始日期和结束日期

虽然两种都支持，但第一版的产品主路径偏 `bounded`，更贴合“在某段时间内持续执行”的阶段性习惯。

## 习惯记录模型

习惯本体和每日执行记录分开。

执行记录至少包含：

- `habitId`
- `date`
- `status`
- `updatedAt`

其中 `status` 第一版只要求支持两种表现：

- `completed`
- `missed`

实现层可以显式存储，也可以部分由界面派生；本规格只约束行为，不约束底层存储实现。

## 执行规则

### 基础规则

1. 第一版只记录 `做了 / 没做`。
2. 不记录次数、分钟数、条数或完成质量。

### 当日默认状态

到了应执行日，如果用户没有进行任何操作，则当天在界面上默认显示为 `未完成`。

这不是“空白未生成”，而是可见的未完成状态。

### 回补规则

1. 用户之后可以把某个 `未完成` 的执行日改成 `已完成`。
2. 不限制只能补最近 1 天或 7 天。
3. 只要该日期仍然处在该习惯的有效周期内，就允许回补。

这条规则是为了贴合“经常第二天统一补勾”的真实使用方式。

## 归档规则

第一版不做 `暂停`，只做 `归档`。

`归档` 的语义：

- 这个习惯以后不再继续执行。
- 但历史记录保留。

归档后的行为：

1. 不再出现在 `今天该做`。
2. 不再生成新的待执行日期。
3. 仍可在 `全部习惯` 中看到。
4. 旧的完成记录保留。

## 与项目系统的边界

以下行为第一版明确禁止：

1. 把习惯显示在项目待办清单中。
2. 把习惯自动转换为项目任务树节点。
3. 把项目任务自动沉淀成习惯。

原因是：

- 项目系统解决的是“交付物推进”。
- 习惯系统解决的是“重复行为维持”。

两者如果混在一起，会把现有已经跑顺的项目推进模型搞脏。

## 第一批候选习惯

当前比较适合放入第一版习惯系统的候选：

1. `爬坡`
2. `用 Pocket 3 记录当天`
3. `看 AI HOT 日报`

`整理 GitHub Star Top` 暂时保留观察：

- 如果只是每天轻扫一下，可以作为习惯。
- 如果目标是沉淀出明确素材产出，更适合继续作为任务或项目处理。

## 成功标准

这版习惯系统如果要算成功，至少要满足：

1. 用户能把重复行为从项目系统中分离出去。
2. 用户打开 `习惯` 页面时，能立刻看见今天该做什么。
3. 用户漏掉后还能回补，不会因为系统太硬而放弃使用。
4. 习惯系统不会像复杂任务工具那样变成新的维护负担。

## 后续观察点

第一版上线后，需要继续观察这些问题，再决定后续是否扩展：

1. `整理 GitHub Star Top` 这种边界模糊项，究竟更像习惯还是任务。
2. 用户是否真的需要“连续打卡”或数量统计。
3. 用户是否需要把 `今天` 视角进一步细分为更强的执行面板。
